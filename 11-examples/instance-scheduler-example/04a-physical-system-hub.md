# Step 4a — Physical-system ADR (hub)

## Purpose

Describe **deployable hub topology**: DynamoDB data layer, Lambda orchestration suite, EventBridge buses, SNS/log fan-out, and **regional** scope. Maps to CDK entry points under `source/instance-scheduler/lib/` in the **[upstream repository](https://github.com/aws-solutions/instance-scheduler-on-aws)** (for example `core-scheduler.ts`, `instance-scheduler-data-layer.ts`).

> **Illustrative only.** Pedagogical stub; **ste-spec** is normative.

## Illustrative excerpt

```yaml
adr_type: physical-system
id: ADR-PS-INST-001
title: Instance Scheduler — hub physical topology

implements_logical:
  - ADR-L-INST-001

context: |
  Hub stack deploys the control plane in a chosen AWS account/region set. Core scheduler wires Lambdas,
  DynamoDB tables, EventBridge buses, and observability helpers generated by CDK.

technology_stack:
  - category: infrastructure
    name: AWS CloudFormation_via_CDK
    version: solution-defined
    rationale: Matches AWS Solutions packaging model.

system_boundaries:
  - id: SYSBOUND-5181
    name: Hub control plane
    description: >-
      Encapsulates schedule evaluation, API calls to EC2/RDS (and related services), persistence,
      and cross-region/event routing owned by the hub stack.
    external_dependencies:
      - Amazon EC2 / RDS control APIs
      - AWS Lambda / EventBridge / DynamoDB / SNS / KMS
    exposed_interfaces:
      - Hub configuration and schedule maintenance (CLI + console workflows)

external_systems:
  - id: EXT-5181
    name: Amazon EC2
    classification: aws_service
  - id: EXT-5182
    name: Amazon RDS
    classification: aws_service

component_topology:
  components:
    - name: Orchestration Lambdas
      type: scheduler
      purpose: Periodic scheduling, request handling, registration, metrics helpers.
      implements_adr: ADR-PC-INST-001
    - name: DynamoDB data layer
      type: database
      purpose: Config, state, maintenance windows, resource registry.
      implements_adr: ADR-PC-INST-002
    - name: Operator CLI
      type: library
      purpose: Maintain schedules and configuration from workstations/CI.
      implements_adr: ADR-PC-INST-003
  relationships:
    - from: Orchestration Lambdas
      to: DynamoDB data layer
      type: reads_from
    - from: Orchestration Lambdas
      to: DynamoDB data layer
      type: writes_to
    - from: Operator CLI
      to: DynamoDB data layer
      type: writes_to
      protocol: aws_api
      description: CLI uses AWS APIs to mutate configuration artifacts per solution design.

deployment_topology:
  platform: aws
  regions: customer_selected_multi_region
  notes: >-
    Regional EventBridge bus naming follows IS-LocalEvents-${Namespace}; global bus IS-GlobalEvents-${Namespace}
    (see core-scheduler.ts in upstream source).
```

## What to read from it

- **SYSBOUND-5181** is the **whiteboard** you would draw next to the **hub CloudFormation stack** in the AWS console.
- **component_topology** points forward to **three** physical-component ADRs for higher fidelity than the AI Gateway single-PC step.

## What this step produced and why it matters

The **hub** physical-system ADR turns logical commitments into a **deployable hub footprint**: boundaries, major **components**, and **relationships** you could sketch next to the **hub stack** in AWS. **Full** topology for the example—including **spoke** trust and registration—is completed only after [Step 4b](./04b-physical-system-remote.md). This step’s output is the **central** side of that story: what runs in the **hub account** and how it expects to **consume** remote trust and data.
